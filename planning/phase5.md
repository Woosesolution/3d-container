# Phase 5 - 多语言增强与本地化质量收敛

## 1. 目标

- 在 Phase 1 演示级中英双语基础上，补齐完整产品级多语言能力。
- 实现“选择语言后，界面对话与输出均为对应语言”。
- 确保默认语言为中文（`zh-CN`）。

## 2. 范围

### 2.1 In Scope

- 语言支持范围：
  - `zh-CN`（默认，Phase 1 已具备）
  - `en`（Phase 1 已具备）
  - `fr`
  - `de`
- 语言切换与持久化：
  - 未登录用户使用本地存储记忆语言
  - 已登录用户使用 `profiles.preferred_locale`
- 文案接入：
  - 补齐 Phase 1 未覆盖的 UI 文案与边界提示
  - 系统提示（成功/失败/校验）全量覆盖
  - 自动装柜失败原因与解释信息全量覆盖
  - 历史记录页与摘要输出全量覆盖
- 回退链路：
  - 用户手动选择 > `preferred_locale` > 浏览器语言 > 默认 `zh-CN`
- 测试与质量：
  - 单元测试、集成测试、E2E 多语言主链路
  - key 完整性检查（缺失翻译检测）

### 2.2 Out of Scope

- 演示级中英双语能力重做（Phase 1 已交付，不在本 phase 重复建设）
- 机器自动翻译流水线
- 术语审校平台与专业翻译管理系统
- 本地化图片/语音等非文本资产

## 3. 设计约束

- 统一使用 `t(key, params)` 输出文案，禁止新硬编码文本。
- 失败原因先产出稳定 `reason_code`，再映射为多语言文案 key。
- 数值、单位、日期格式按 locale 输出，但业务计算逻辑不受语言影响。

## 4. 代码落点

```txt
src/
  app/
    locale.js
  i18n/
    index.js
    locales/
      zh-CN.json
      en.json
      fr.json
      de.json
  domain/
    reason-codes.js
```

## 5. 交付物

1. 多语言框架接入完成（含回退链路）。
2. 四种语言文案资源文件（`zh-CN/en/fr/de`）。
3. 对话、提示、结果输出的全链路多语言支持。
4. 多语言测试报告与缺失 key 报告。

## 6. 测试清单

1. 四种语言下同一主流程均可完成（含 Phase 1 核心链路）。
2. 切换语言后 UI 文案即时更新。
3. 自动装柜失败原因在四种语言下语义一致。
4. 切换语言不影响装载率与算法结果。
5. 缺失 key 时触发回退，不出现空白文案。

## 7. 通过标准（Gate）

- 四种语言主链路测试全部通过。
- 不存在高优先级本地化缺陷（错语种、空文案、关键误译）。
- 中英文法德在关键业务提示上的语义一致性通过评审。
- 默认中文（`zh-CN`）在未设置场景下稳定生效。

## 8. 风险与对策

- 风险：历史文案硬编码导致漏改。
  - 对策：引入扫描规则，CI 阻断新增硬编码。
- 风险：翻译 key 命名混乱导致难维护。
  - 对策：按模块命名空间管理 key，并维护词汇表。
- 风险：不同语言文案长度差异导致 UI 挤压。
  - 对策：关键页面做自适应布局与截断策略测试。
