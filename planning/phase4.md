# Phase 4 - 全量集成、回归与发布

## 1. 目标

- 将前 3 个 phase 的能力稳定整合为可交付版本。
- 完成端到端回归、文档与发布准备。

## 2. 范围

### 2.1 In Scope

- 全链路联调：
  - 登录 -> 选择租户 -> 产品录入 -> 托盘计算 -> 自动/手动装柜 -> 保存历史 -> 回看历史
- 质量治理：
  - 回归测试清单固化
  - 错误处理与提示统一
  - 关键路径性能优化
- 文档与交付：
  - 部署说明
  - 运维说明（账号、密钥、备份）
  - 常见故障排查

### 2.2 Out of Scope

- 大规模商业化监控体系
- 企业级 SSO/审计合规扩展
- 原生移动端应用

## 3. 端到端场景

1. 新用户注册并加入租户。
2. 创建产品参数并生成托盘计划。
3. 执行自动装柜并手动微调。
4. 保存 load run 并在历史中恢复。
5. 另一租户登录后验证数据隔离。

## 4. 测试体系

### 4.1 回归测试

- Phase 1 算法与交互能力不得回归。
- Phase 2 多租户隔离不得回归。
- Phase 3 评分与解释能力不得回归。

### 4.2 安全测试

- 越权读取/写入测试。
- 鉴权失效与会话过期测试。
- SQL 注入与参数边界测试（通过 SDK 也要验证输入）。

### 4.3 性能测试

- 大量产品场景下渲染稳定性。
- 自动装柜耗时与交互响应时间。

## 5. 交付物

1. 集成版应用（可部署）。
2. 回归测试报告。
3. 安全测试报告。
4. 部署与运维文档。

## 6. 发布通过标准（Final Gate）

- P0/P1 缺陷清零。
- 端到端关键路径全部通过。
- 多租户隔离与权限验证通过。
- 性能达到团队约定阈值。

## 7. 上线后观测建议

- 记录关键事件：
  - 登录失败率
  - 自动装柜失败原因分布
  - 平均装柜耗时
- 每周复盘 `debug-history`，持续修正规则和策略。

## 8. 风险与对策

- 风险：跨模块耦合导致集成期问题集中爆发。
  - 对策：先做 smoke test，再做全量回归。
- 风险：免费层配额触顶。
  - 对策：告警阈值前置，准备升级或迁移预案。
